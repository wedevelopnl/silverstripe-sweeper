Veilig database­schema opruimen
zonder CREATE DATABASE

We nemen het schema op dat SilverStripe zélf zou bouwen, zonder ook maar één database aan te raken.

Module: wedevelopnl/silverstripe-sweeper  ·  Datum: 2026-06-02  ·  Status: geverifieerd op host-project (dry-run + A/B)
Update na deze pitch: de oude sweeper-artefacts is alsnog gedeprecate en verwijderd. sweeper-schema-artefacts vervangt hem. Slides die zeggen dat de oude taak "blijft", beschrijven het plan zoals het op 2026-06-02 was.

De aanleiding

De bestaande taak loopt stuk op productie

SweeperArtefactsTask vergelijkt het live-schema met een "schoon" schema om verweesde tabellen, kolommen en indexen te vinden en te verwijderen. Dat schone schema wordt gebouwd met SilverStripe's TempDatabase.

Symptoom

Op managed/cloud hosting krijgt de app-DB-gebruiker een permissiefout. De taak kan niet draaien.

Wat we dachten

"Het bouwt toch een in-memory database?" De beschrijving van de taak zegt dat letterlijk.

Die aanname klopt niet. En dat is de kern van het probleem.

Root cause

TempDatabase maakt een echte database aan

// TempDatabase::build() -> Database::selectDatabase(create: true)
//   -> MySQLSchemaManager::createDatabase()
CREATE DATABASE "ss_tmpdb_1717322000_4821993" ...

Een fysieke CREATE DATABASE op dezelfde server als productie. Vereist server-rechten die de app-user niet (meer) heeft.

Geen regressie

De TempDatabase-code is byte-identiek tussen SS 4.13 en 5.2. Niet het framework veranderde, de hosting werd strenger.

"In-memory" = alleen SQLite

Voor MySQL/MariaDB bestaat zo'n database niet; het was altijd een echte temp-database.

Het sleutelinzicht

dev/build en TempDatabase gebruiken dezelfde build

Beide produceren het schema via exact dezelfde methode:

TableBuilder::singleton()->buildTables($schema, $dataClasses, ...);
  // -> per class: requireTable()  +  elke extensie's augmentDatabase()
  // -> schemaUpdate() buffert, en flusht aan het eind via createTable()
De database eronder is enkel het rendertarget. De waarheid zit in de build-traversal, niet in de engine. Dus: als we die traversal draaien en createTable() onderscheppen, hebben we het complete schema zonder database.

De oplossing

Een recording schema manager

We installeren tijdelijk een opnemende schemamanager als de actieve manager, draaien de build, en registreren wat er gebouwd zou worden in plaats van SQL uit te voeren.

✓ Geen database

Geen temp-DB, geen CREATE DATABASE.

✓ Geen dependency

Geen silverstripe/sqlite3 nodig.

✓ Aparte taak

De bestaande sweeper-artefacts blijft volledig intact.

Nieuw: sweeper-schema-artefacts (segment), met RecordingSchemaManager + SchemaDiff.

Hoe het werkt

Onderschep createTable()

// RecordingSchemaManager extends MySQLSchemaManager
public function createTable($table, $fields, $indexes, ...) {
    // registreer i.p.v. SQL uitvoeren
    $this->recorded[$table] = [
        'columns' => array_keys($fields),   // incl. ID
        'indexes' => $indexes,              // incl. fulltext-type
    ];
    return $table; // geen DDL
}

tableList/fieldList/indexList geven leeg terug, zodat élke vereiste als "create" wordt opgenomen. Anders zou een bestaande kolom niet gebufferd worden en onterecht verweesd lijken.

Tabellen/kolommen op naam, indexen op signatuur.

→
4

Op de echte connectie. Standaard dry-run.

De echte connectie wordt nooit losgekoppeld; in finally gaat de echte schemamanager altijd terug. Geen tweede DB, geen reconnect-hack.

Indexen

Vergelijk op signatuur, niet op naam

Indexnamen kunnen variëren; de kolomcompositie niet. Daarom vergelijken we op type + gesorteerde kolommen.

# MySQL (huidig)      signatuur
idx_old_feature  ->  index:OldColumn     # niet in opname -> verweesd
URLSegment       ->  unique:URLSegment  # matcht -> blijft

index:OldColumn  ->  DROP INDEX "idx_old_feature"  (op echte naam)
Bonus t.o.v. de SQLite-aanpak: omdat de opname vóór de render zit, behouden fulltext/hash/rtree hun type native. De eerder bedachte metadata-hybride is niet meer nodig.

Veiligheid

Vijf vangnetten

🎫 Bevestigingstoken

De dry-run print een token (hash van de droppable-set). run=yes weigert zonder of met verouderd token: je voert exact de gereviewde set uit, of niets.

👁 Dry-run default

Pas met run=yes token=... worden DROP's uitgevoerd.

🔒 PRIMARY beschermd

De primaire sleutel wordt nooit gedropt, in de diff én in de drop.

🛑 Abort-on-error

Faalt de opname, dan stopt de taak zonder wijzigingen. Een onvolledig referentieschema is onveilig om te diffen.

✅ Zelf-validerend

Op een net ge-dev/build-te DB hoort de dry-run leeg te zijn. Plus telling: N reference tables recorded.

Toegang: /dev/tasks is in dev-modus open, op test/live alleen met ADMIN/ALL_DEV_ADMIN/BUILDTASK_CAN_RUN. Het token is geen geheim maar een review-binding.

De beste verificatie

Zelf-validerende eigenschap

Omdat de recorder exact dezelfde traversal draait als dev/build, geldt op een schone (net gebouwde) database:

referentieschema == gebouwd schema   =>   diff is leeg
Verschijnt er tóch iets op een schone DB, bijvoorbeeld een _Versions-tabel, een many_many-tabel of de ID-kolom, dan heeft de opname iets gemist en is er een bug. Dit is je betrouwbaarste check vóór je ooit run=yes draait.

Wat er gebouwd is

Scope

Samenvatting

Waar we staan

  • Probleem: TempDatabase maakt een echte database; faalt zonder server-rechten.
  • Inzicht: dev/build bouwt via TableBuilder; de DB is enkel rendertarget.
  • Oplossing: recording schema manager, geen database, geen dependency, als aparte taak.
  • Indexen: op signatuur; fulltext native correct.
  • Vertrouwen: A/B-geverifieerd op Olympia (identieke uitkomst als de oude taak), bevestigingstoken, dry-run default, abort-on-error.
Volgende stap: committen op een branch; eventueel een gecontroleerde run=yes op een wegwerp-database.
